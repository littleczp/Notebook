# AI

## **不和模型对赌**

加多脚手架也许能多拿一点效果，但下一代模型可能直接"白送"；所有非模型能力最终都会变成技术债。



**什么是"脚手架"（Scaffolding）？**

> 脚手架是为了弥补当前模型能力的不足，而编写的**额外代码、复杂的提示词工程（Prompt Engineering）、逻辑链（Chain of Thought）、或者外挂的规则系统**。
>
>
>
> 脚手架例子： 现在的模型数学不好，你写了一套复杂的 Python 代码来拦截用户的数学问题，调用计算器，再把结果塞回给模型。



**什么是"和模型对赌"？**

> 你假设模型的某些缺陷（如幻觉、逻辑差、上下文短）是长期的，并把你的核心竞争力建立在"修补这些缺陷"上
>
>
>
> 对赌失败的例子： 很多人曾花费巨大精力开发"长文本摘要切片算法"，因为 GPT-3.5 的上下文窗口很小。结果 GPT-4-Turbo 和 Gemini 1.5 Pro 出来后，支持超长上下文，那些切片算法瞬间变得毫无意义。



## **迭代速度**

快速试错比"设计完美方案"更重要。



**产品功能**

> **功能不是规划出来的，是从用户行为里"长"出来的。**
>
>
>
> 不要教育用户改变行为，而是顺着他们已经发生的行为去放大它。



**产品认知**

> **Agent 的能力边界，会每几个月重写一次**。
>
>
>
> 对“它能不能做”的判断很快会过时，工程师必须不断重置认知。
>
> 多 agent 协作，是能力放大的关键变量。并行 sub-agent、本质上是 test-time compute 和上下文隔离的组合，会显著提升复杂任务能力。
